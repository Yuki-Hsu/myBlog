.gitignore 文件无效

发表于 2020-04-22 分类于 Git 学习

记一次 .gitignore 文件无效的解决办法

在 git 仓库中，如果想忽略某个文件或文件夹，使其不被提交到 git 版本库中去，可以在仓库根目录中添加 .gitignore 文件。但在有些时候，修改了 .gitignore 文件，却发现无效。

原因

.gitignore 只能忽略那些原来没有被跟踪的文件，如果某些文件之前已经被纳入了版本管理中，则修改 .gitignore 是无效的。

解决办法

详解

下面这个图展示了工作区、版本库中的暂存区和版本库之间的关系：

当对工作区修改（或新增）的文件执行 git add 命令时，暂存区的目录树被更新，同时工作区修改（或新增）的文件内容被写入到对象库中的一个新的对象中，而该对象的 ID 被记录在暂存区的文件索引中。

当执行提交操作 git commit 时，暂存区的目录树写到版本库（对象库）中，master 分支会做相应的更新。即 master 指向的目录树就是提交时暂存区的目录树。

当执行 git reset HEAD 命令时，暂存区的目录树会被重写，被 master 分支指向的目录树所替换，但是工作区不受影响。

当执行 git rm --cached <file> 命令时，会直接从暂存区删除文件，工作区则不做出改变。

当执行 git checkout . 或者 git checkout -- <file> 命令时，会用暂存区全部或指定的文件替换工作区的文件。这个操作很危险，会清除工作区中未添加到暂存区的改动。

当执行 git checkout HEAD . 或者 git checkout HEAD <file> 命令时，会用 HEAD 指向的 master 分支中的全部或者部分文件替换暂存区和以及工作区中的文件。这个命令也是极具危险性的，因为不但会清除工作区中未提交的改动，也会清除暂存区中未提交的改动。

git 命令

git rm Remove files from the working tree and from the index

  • git rm <file> 默认删除工作区和暂存区的文件
  • 使用选项 -r 递归地删除目录及内容
  • 使用选项 --cached 只删除暂存区里的文件

git restore Restore working tree files

  • git restore <file> 默认恢复工作区的文件，当工作区的文件被意外删除后可以使用
  • git restore -W <file> or git restore --worktree <file> 恢复工作区文件
  • git restore -S <file> or git restore --staged <file> 恢复暂存区文件

If not specified, the default restore source for the working tree is the index, and the default restore source for the index is HEAD. When both --staged and --worktree are specified, --source must also be specified.
